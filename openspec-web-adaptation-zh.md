# OpenSpec：ZType LibGDX Web 版接入方案（中文草案）

## 1. 目标

在保持现有 `core` 单一玩法逻辑的前提下，新增 Web 端（浏览器）可运行版本，优先实现最小可玩（MVP）。

核心原则：

- 一套 `core` 逻辑复用。
- 新增 `gwt` 平台模块作为 Web 启动壳。
- 不影响现有 `desktop` 与 `android` 已有可运行能力。

---

## 2. 本次范围

### In Scope

1. 新增 `gwt` 模块并完成 Gradle 接入。
2. 创建 Web 启动器（EntryPoint）并可加载 `ZTypeGame`。
3. 提供可本地访问的开发运行方式（superDev / 内置 server）。
4. 浏览器内完成基础可玩：敌人生成、输入、计分、重开。

### Out of Scope

1. 线上部署（CDN、域名、HTTPS、缓存策略）。
2. 高级 Web 兼容优化（低端浏览器专项适配）。
3. 音频与资源管线全面优化。

---

## 3. 架构设计

- `core`：继续放游戏逻辑、渲染、输入统一入口。
- `desktop` / `android`：保持不变。
- `gwt`（新增）：
  - GWT 启动类
  - HTML 容器页
  - GWT 编译配置

输入策略：

- Web 键盘输入沿用 `onLetterInput(char)`。
- 触屏按钮逻辑继续复用（移动浏览器可点按）。

---

## 4. 风险与应对

1. GWT 构建配置复杂度高于 desktop/android。
   - 应对：先跑通最小模板，再逐项补配置。
2. 浏览器字体显示与平台差异。
   - 应对：先保证可玩，再做表现微调。
3. 首次编译时间较长。
   - 应对：开发阶段用 superDev，发布前再全量编译。

---

## 5. 验收标准（DoD）

1. `gwt` 模块可编译。
2. 本地打开 Web 页面可进入游戏。
3. 键盘输入可打字击毁敌人。
4. 基础 HUD、暂停/重开可用。
5. 不影响 desktop/android 现有构建通过。

---

## 6. 待确认项

1. Web 首版是否只做桌面浏览器支持？（建议：是）
2. 是否需要同一页面保留触屏按钮？（建议：保留，方便移动浏览器）
3. Web 输出目录是否沿用默认（建议：是）
